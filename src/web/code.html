Code walkthrough

What BIP-110 actually changes

BIP-110 is deliberately tiny — in Bitcoin Knots it is 5 modified and 72 added lines across 5 source files, small enough to read end to end. It promotes a set of existing standardness filters into consensus rules that reduce how much arbitrary data a transaction can commit on-chain. Here are the seven rules and how each is enforced.

The seven rules

While BIP-110 is active, script validation runs with one new flag, SCRIPT_VERIFY_REDUCED_DATA, which swaps in tighter limits. Each rule is a consensus check that rejects a specific kind of oversized or arbitrary data.

1

Output size limit

A new output's script may not exceed 34 bytes — enough for standard payment outputs. The one exception is a data output (OP_RETURN), allowed up to 83 bytes. Enforced by a new CheckOutputSizes function.

2

Data-push limit

Individual data pushes and witness stack items may not exceed 256 bytes (down from the usual 520), via MAX_SCRIPT_ELEMENT_SIZE_REDUCED. A BIP16 P2SH redeemScript push is exempted so existing P2SH keeps working.

3

Undefined witness versions

Spending an output with an undefined witness version becomes invalid (creating one stays valid). This also closes a Bitcoin Core edge case around anchor outputs.

4

Taproot annex rejected

A witness stack that carries a Taproot annex is invalid. Previously the annex was merely non-standard policy; BIP-110 makes rejecting it a consensus rule.

5

Control-block size limit

A Taproot control block larger than 257 bytes is invalid (TAPROOT_CONTROL_MAX_SIZE_REDUCED) — still enough for a merkle tree of 128 script leaves.

6

No OP_SUCCESS

A tapscript that contains an OP_SUCCESS opcode anywhere is invalid, closing off a large class of otherwise-unconstrained script data.

7

No OP_IF / OP_NOTIF

A tapscript that executes OP_IF or OP_NOTIF is invalid, reusing the existing minimal-IF error as a stricter form of the same restriction.

How it stays backwards-compatible

Grandfathering, per input

Nothing already on the chain is retroactively invalidated. Rules apply per input: each input is judged against the block height of the output it spends. Inputs spending outputs created before activation have the reduced-data flags cleared individually (flags_per_input in CheckInputScripts, gated on reduced_data_start_height). Because flags now vary per input, the script-verification cache is bypassed when that happens.

Where it's enforced

Most checks live in the interpreter behind SCRIPT_VERIFY_REDUCED_DATA. The output-size rule runs in CheckTxInputs (not CheckTransaction, which can't see deployment status), and the coinbase/generation transaction — which has no inputs — gets an explicit CheckOutputSizes call so Rule 1 applies chain-wide.

Activation & expiry — a temporary soft fork

BIP-110 extends the BIP9 version-bits state machine with BIP8-style mandatory signalling and a new terminal state, so the fork is guaranteed to activate and then automatically lapses.

Mandatory signalling

In the retarget period(s) just before the deadline (max_activation_height), blocks that do not set version bit 4 are rejected as invalid (DeploymentMustSignalAfter, checked by a new ContextualCheckBlockHeaderVolatile). One period before the deadline, the state is forced to LOCKED_IN regardless of the signalling percentage — so lock-in is guaranteed no later than the deadline.

Automatic expiry

A new active_duration (52,416 blocks, ~1 year) makes the deployment temporary. Once activation_height + active_duration is reached, the state machine moves from ACTIVE to a new terminal EXPIRED state (DeploymentActiveAfter): the reduced-data rules stop being enforced and outputs are unrestricted again.

The activation threshold is a per-deployment 55% of a 2016-block difficulty period (1109 blocks), overriding the usual 95% — a small change to the Threshold logic in versionbits.cpp.

The five files

src/consensus/tx_verify.cppNew CheckOutputSizes — Rule 1 (output and data-carrier size limits).
src/script/interpreter.cppReduced element & control-block sizes, annex rejection, OP_IF/OP_NOTIF, the P2SH redeemScript exemption, and the anchor-output fix — Rules 2, 4, 5, 7 (and 3).
src/validation.cppPer-input grandfathering (flags_per_input), the coinbase output check, and block-header mandatory-signalling checks.
src/deploymentstatus.hDeploymentActiveAfter (temporary-fork expiry) and DeploymentMustSignalAfter (mandatory-signalling window).
src/versionbits.cppForced LOCKED_IN before the deadline, the ACTIVE → EXPIRED transition, and the per-deployment 55% threshold.

Why it's reviewable

The point of keeping BIP-110 to a few dozen lines is that you don't have to be a C++ developer to audit it. Most rules simply promote limits nodes already apply as policy into consensus, guarded by a single flag. The per-input grandfathering and the temporary-expiry state are the only genuinely new machinery.
