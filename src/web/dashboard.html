Miner signalling for BIP-110 (block version bit 4)

This is the authoritative figure. BIP-110 activates when 55% (1109) of the 2016 blocks in a difficulty adjustment period set bit 4 in their version field — the tally is counted per retarget period, not over a rolling window. Measured by scanning the current period's block headers from your own node — not from peer gossip.

Road to mandatory lock-in

BIP-110 uses BIP8-style mandatory signalling in the retarget period before a guaranteed lock-in — during that window, blocks that don't signal bit 4 are rejected as invalid. Being a temporary soft fork, it then expires ~1 year (52,416 blocks) after activation, and the data limits are lifted. The countdown is estimated from the current chain tip at ~10 min/block. The milestone heights below assume the guaranteed-latest path (mandatory lock-in at 963,648); if miners reach 55% signalling in an earlier period, lock-in — and every date after it — arrives sooner.

Chain integrity

Mandatory signalling is the moment a split becomes possible: a node enforcing BIP-110 rejects blocks that don't set bit 4, so if miners don't signal, enforcing and non-enforcing nodes follow different chains. This watches for exactly that — from getchaintips on your own node, corroborated by the tip heights peers report during the crawl. Ordinary orphan races produce short branches that resolve, so they are not counted.
Caveat: getchaintips is authoritative only for branches your node accepted and then rejected. Mandatory signalling is checked at the header level, so a chain your node refuses outright may never appear there at all — it would simply stall. For the definitive test, the chain view asks every reachable peer which block it has, and compares hashes.
