Signal vs noise in the address book

The big number is addresses, not machines — there are nowhere near that many nodes out there. getaddr doesn't return a peer's live connections; it returns its whole address book — every address it has heard of over months — and we take the union of that across thousands of peers. Two things inflate it hugely: dead nodes are remembered and keep circulating through gossip long after they're gone, and one machine emits many addresses over time. That second effect dominates, and this crawler's own data shows it: 83% of the never-answering IPv6 addresses share a /64 — a single host's own subnet, which privacy addressing rotates constantly (one /64 here accounts for ~2,900 addresses on its own) — and 72% of the IPv4 ones share a /24, typical of dynamic ISP ranges and Tor exit pools.
So read it as "every address ever gossiped to us, mostly the same hosts over and over", not as a node count. The reachable figure is the real one. Each dead address still costs a full connection timeout to rule out, which is why the crawler can look busy while finding almost nothing new.

Client versions over time

Hourly population of reachable nodes per client version (e.g. Bitcoin Knots 29.3.0), recorded by the crawler as it runs — so you can watch releases roll out and old ones drain away. Only the 10 most common versions are plotted; the network runs about a hundred, but the rest are a long tail of a handful of nodes each. Lines are coloured by client — Knots orange, Core blue — with lighter shades separating versions of the same client. Hover a line for its version and node count. This series starts from the first run of the build that added it and grows from here.
