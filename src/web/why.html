Reduced Data Temporary Softfork

Why support BIP-110?

BIP-110 is a temporary, miner-activated soft fork that tightens Bitcoin's limits on non-monetary data. The goal is simple: keep the blockchain lean enough that ordinary people can keep running full nodes — and keep Bitcoin's scarce block space working for money.

What it actually does

Bitcoin blocks are a shared, permanent, every-node-forever resource. Over time, techniques emerged to stuff arbitrary data — images, tokens, and other non-financial payloads — into that space. Every byte is downloaded, validated, and stored by every full node on Earth, essentially forever.

BIP-110 restores tighter limits on that arbitrary-data usage. It activates like BIP-9: miners set bit 4 in the block version, and once 1109 of any 2016 blocks (55%) signal, the rules lock in. Because it is temporary and soft, it is low-risk and does not split the network — nodes that don't upgrade still follow the chain.

The case for support

Six reasons node operators, miners, and businesses back BIP-110.

🌐

Protect decentralisation

A leaner chain means cheaper disk, bandwidth, and sync time. The lower the cost of running a full node, the more people can — and node count is the backbone of Bitcoin's censorship resistance.

💸

Bitcoin is money first

Block space is scarce and paid for by everyone. Reserving it for monetary transactions keeps fees predictable for real payments instead of bidding wars over arbitrary data storage.

🧹

Curb spam & bloat

Non-financial payloads inflate block and chain size without adding monetary utility. Tighter data limits slow that growth and keep the UTXO set and archive healthier.

↩️

Low-risk & temporary

It's a soft fork (backwards-compatible) and explicitly temporary. It can lapse or be revisited — no contentious hard fork, no forced chain split, no permanent commitment.

📊

Activated by the market

Nothing is imposed. Activation requires 55% of a difficulty period's 2016 blocks to signal — a genuine, measurable supermajority of hash power choosing the rule, not a top-down decree.

🗳️

You decide

Support is opt-in: run software that enforces the limits and, if you mine, signal for it. This crawler exists so anyone can watch that choice play out across the network in real time.

How to support it

  1. Run a BIP-110-ready node. Use a Bitcoin Knots build that ships BIP-110 — a mainline build dated Knots:20260508 or later, or any build tagged +bip110 / UASF-BIP110.
  2. Tighten your data policy. Add these to bitcoin.conf and restart: 
    datacarrier=0
permitbaremultisig=0
rejectparasites=1
rejecttokens=1
  3. Signal, if you mine. Configure your template so blocks set version bit 4. Every signalling block moves the network toward the 55% threshold.
  4. Check yourself on the map. Open the live crawler — reachable nodes running a ready build show as BIP-110 ready.

An honest note

This is a contested topic. Reasonable people disagree about on-chain data limits — some argue fees alone should ration block space, or that filters are easy to bypass. BIP-110's temporary, opt-in, miner-signalled design is a deliberate response: it lets the market decide with minimal risk. Read the specification and judge for yourself before you signal.
