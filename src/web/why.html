Reduced Data Temporary Softfork

Why support BIP-110?

BIP-110 is a temporary, miner-activated soft fork that tightens Bitcoin's limits on non-monetary data. The goal is simple: keep the blockchain lean enough that ordinary people can keep running full nodes — and keep Bitcoin's scarce block space working for money.

What it actually does

Bitcoin blocks are a shared, permanent, every-node-forever resource. Over time, techniques emerged to stuff arbitrary data — images, tokens, and other non-financial payloads — into that space. Every byte is downloaded, validated, and stored by every full node on Earth, essentially forever.

BIP-110 restores tighter limits on that arbitrary-data usage. It activates like BIP-9: miners set bit 4 in the block version, and once 1109 of any 2016 blocks (55%) signal, the rules lock in. Because it is temporary and soft, it is low-risk and does not split the network — nodes that don't upgrade still follow the chain.

The case for support

The case node operators, miners and businesses make for BIP-110.

🌐

Verification must stay cheap

Every byte in a block is downloaded, validated and stored by every full node on Earth, permanently. Arbitrary data raises that cost for everyone and returns nothing to Bitcoin's monetary function. The cost of running a node is the ceiling on how many people can verify Bitcoin for themselves — and self-verification is the entire point of the system.

💸

Block space is for money

Block space is Bitcoin's scarcest resource and the one thing that cannot be replaced. Cheap, permanent, replicated storage exists everywhere else. When block space is auctioned to image storage, ordinary payments bid against a use case that never needed a censorship-resistant monetary ledger in the first place.

🧹

The UTXO set never forgets

Inscription and token schemes mint economically dead dust outputs — outputs nobody will ever profitably spend. Every one is tracked by every validating node for as long as it stays unspent. This is the one form of bloat that cannot be pruned, and it is the growth that most directly raises the hardware floor for running a node.

↩️

Low-risk and self-expiring

It is a soft fork — backwards-compatible, no chain split, non-upgraded nodes keep following the chain. And it is explicitly temporary: the rules lapse automatically about a year after activation. If it proves wrong, it undoes itself — a far smaller commitment than the status quo, which changed by client default and has no expiry at all.

📊

Decided by the market, not a release note

Nothing is imposed. Activation needs 55% of a difficulty period's 2016 blocks to signal — a measurable supermajority of hash power actively choosing the rule. Compare that with how the limits were removed: a default changed in a software release.

🗳️

Your node is your vote

Support is opt-in and verifiable. Run software that enforces the limits, tighten your own data policy, and — if you mine — signal bit 4. This crawler exists so that choice is measurable in public, in real time, rather than merely argued about.

How we got here

BIP-110 is not a new restriction on Bitcoin. It restores one that held for nearly a decade, and closes the specific techniques that got around it.

2014–2022 · Data was always discouraged

Bitcoin Core shipped data-carrier limits from 2014: a cap on OP_RETURN payloads (settling at 83 bytes) and limits on bare multisig. Putting arbitrary data on-chain was never impossible — it was deliberately expensive and inconvenient, and that was the point. But these were standardness policy, not consensus: a brake any miner could ignore, not a wall.

Jan 2023 · Ordinals bypasses every limit at once

Inscriptions place arbitrary payloads inside a tapscript witness, wrapped in an OP_FALSE OP_IF … OP_ENDIF envelope that never executes. That sidestepped the data limits entirely and paid the SegWit witness discount — roughly 4× cheaper per byte than ordinary transaction data. Taproot had already lifted the script-size ceiling, so payload size was effectively unbounded. Within weeks BRC-20 added token minting on top, and with it dust outputs at scale.

Apr 2024 · Runes, and normalisation

Runes launched at the halving using OP_RETURN. In fairness these are UTXO-friendlier — an OP_RETURN output is provably unspendable and never enters the UTXO set — but they still consume permanent block space, and they cemented the idea that Bitcoin's ledger is a general-purpose data bus.

Oct 2025 · Core v30 removes the brake

Bitcoin Core v30 removed the default OP_RETURN size limit, making large data outputs relayable by default. The last standing policy brake came off — and it came off through a change of defaults in a software release, not through any decision by the network. BIP-110 is the network's reply: put the limits back where a release note cannot move them — in consensus, and only for as long as the network votes to keep them.

The technical case

What each rule actually closes, and why policy alone stopped working.

The witness discount is being arbitraged

SegWit prices witness bytes at 1 weight unit instead of 4. That discount exists to make signature data cheaper and encourage UTXO-efficient spending — it was never priced for bulk storage. Taproot then removed the script-size ceiling for tapscript. Together they made the witness the cheapest permanent storage in Bitcoin. Rule 2 (256-byte push limit) and rule 7 (no OP_IF in tapscript) close that path directly.

Unexecuted data is still stored data

The inscription envelope is OP_FALSE OP_IF — the branch never runs. The payload does nothing, proves nothing and is validated by no one; it exists purely to ride along inside a witness every node must keep forever. Rule 7 makes reaching OP_IF in tapscript invalid at all, which is why it is the single most effective rule in the set.

The UTXO set is the part you can't prune

Block data can be pruned; the UTXO set cannot. Every unspent output must be tracked by every validating node to check the next block. Dust outputs minted by token schemes are rarely worth more than the fee required to spend them, so they sit there indefinitely. Rule 1 (34-byte outputs, 83 for OP_RETURN) removes the room those schemes need while leaving every standard payment type untouched.

Policy was never binding — consensus is

Relay policy is advice. Any miner can ignore it, and direct-to-miner submission routes around it entirely; that is exactly how large payloads kept landing on-chain while nodes still "filtered" them. BIP-110 promotes the same long-standing limits into consensus for roughly a year, so they bind the whole network rather than only the polite parts of it.

How to support it

  1. Run a BIP-110-ready node. Use a Bitcoin Knots build that ships BIP-110 — a mainline build dated Knots:20260508 or later, or any build tagged +bip110 / UASF-BIP110.
  2. Tighten your data policy. Add these to bitcoin.conf and restart: 
    datacarrier=0
permitbaremultisig=0
rejectparasites=1
rejecttokens=1
  3. Signal, if you mine. Configure your template so blocks set version bit 4. Every signalling block moves the network toward the 55% threshold.
  4. Check yourself on the map. Open the live crawler — reachable nodes running a ready build show as BIP-110 ready.

An honest note

This is a contested topic. Reasonable people disagree about on-chain data limits — some argue fees alone should ration block space, or that filters are easy to bypass. BIP-110's temporary, opt-in, miner-signalled design is a deliberate response: it lets the market decide with minimal risk. Read the specification and judge for yourself before you signal.
